# project-log â€” Narrative Project Management for Solo Coders

`project-log` is a discipline package for solo developers using autonomous coding agents. It keeps
the developer and the agent aligned on **where the project is, what's next, and what's shipped** â€”
without tickets, boards, or team ceremony. The durable record is a single append-only file
(`docs/local/project-log.md`) that the coding agent maintains itself at every milestone.

## When to use this skill

Invoke whenever any of these are true:

- The user asks about project status ("where are we", "what's next", "what's shipped")
- The user is starting a new project or adding project-log to an existing repo
- The user wants to ship, replan, park, or log a milestone
- A session is opening on a repo that has a `docs/PRD.md` + `docs/local/project-log.md` pair
- The user mentions a PRD, completion plan, project log, sprint loop, stage ID, or any slash
  command (`/where`, `/roadmap`, `/idea`, `/bootstrap`, `/replan`, `/ship`, `/log`)

## The plan hierarchy

| Tier | Artifact | Purpose | In git? |
|---|---|---|---|
| 1 | **PRD** | Vision, principles, success measures, maturity scale, roadmap phases | Yes (`docs/PRD.md`) |
| 2 | **Completion plan** | Ordered decomposition of PRD into stages `A1`, `A2`, `B1`â€¦ | No (`docs/local/completion-plan-YYYY-MM-DD.md`) |
| 3a | **Project log** (lifetime) | Append-only chronological milestone record. **Never edited, never pruned.** | No (`docs/local/project-log.md`) |
| 3b | **Per-effort progress** (transient) | Sprint-scoped append-only working notes. Tail = resume cursor. | No (`docs/local/progress/<hash>-<date>-<stage>-<slug>.md`) |
| 3c | **STATE.md** (derived) | Human-readable "right now" snapshot regenerated by `/where`. | No (`docs/local/STATE.md`) |

Stage IDs (`[PHASE][N]` â€” e.g. `A3`, `B1`) are the **mapping token** linking branches, commits,
progress files, and project-log entries back to the PRD.

```
Branch:        feat/a3-readiness-recovery-2026-04-23
Progress file: docs/local/progress/5258b3d-2026-04-23-a3-readiness-recovery.md
Commit:        feat(a3): add readiness self-recovery
Project-log:   2026-04-23T14:20Z  A3  started   branch feat/a3-readiness-recovery-2026-04-23
Ship commit:   ship: a3: readiness self-recovery landed
Project-log:   2026-04-23T19:45Z  A3  shipped   readiness self-recovery â†’ checkpoint-20260423-a3-readiness-recovery
```

## Non-negotiable principles

1. **Narrative over tickets.** State is a living story, not a queue.
2. **Single source of truth per tier.** PRD for vision, completion plan for ordering, project-log
   + per-effort progress for chronology, STATE.md is a derived snapshot.
3. **Mapping tokens everywhere.** Every sprint, branch, progress file, log entry, and commit
   carries the stage ID.
4. **Append-only logs; per-effort files distil to milestones on ship.** The per-effort progress
   file is the sprint log â€” its tail is by definition where the agent left off. On ship, its key
   beats become a single line in the lifetime project-log; the per-effort file is then deleted.
   The project-log is never edited or pruned.
5. **Drift check is mandatory.** Session start and every state transition reconciles STATE.md
   against git reality and against the project-log tail.
6. **The tool is dumb; the plan is smart.** Slice size, commit cadence, and ship criteria live
   in the active sprint plan, not in this skill.
7. **Intake is conversational.** Bootstrap is a dialogue with Claude, not a form.
8. **Auto-trigger over slash-typing.** Every sibling skill is described so Claude picks it up
   from natural phrasing. Slash commands are fast-path shortcuts, not the primary surface.
9. **The coding agent owns milestone capture.** Every meaningful state transition (started /
   shipped / decision / blocker / unblock / scope change) is written to the project-log by the
   agent at the moment it happens â€” not retrospectively, not by the user.

## What to do at session start

Run this check in order before touching any code:

1. Read `docs/local/STATE.md` if it exists.
2. Read the **tail of `docs/local/project-log.md`** (last ~20 entries).
3. Read the tail of the active per-effort progress file (last ~30 lines), if one exists.
4. Run `git status --short`, `git branch --show-current`, `git log --oneline -5`.
5. Reconcile. Mismatch patterns:
   - Current branch differs from the latest project-log `started` entry that hasn't been `shipped` yet â†’ flag
   - Uncommitted changes on a branch the project-log marks as already shipped â†’ flag
   - No progress file exists for the Now sprint, but project-log shows it as still active â†’ flag
   - Project-log latest entry > 3 days old (likely missing milestone capture) â†’ flag
   - STATE.md disagrees with the project-log tail â†’ regenerate STATE.md and warn
6. If any mismatch â†’ surface it and block new work until reconciled or explicitly overridden.
7. If no mismatch â†’ report the resume cursor (last "Next:" line of the active progress file) and continue.
8. If no `docs/PRD.md` exists at all â†’ prompt the user to run `/bootstrap`.

## Commands (sibling skills)

| Command | What it does |
|---|---|
| `/bootstrap` | Conversational PRD intake + scaffold for a fresh project |
| `/where`     | Tactical "you are here" â€” reconciles STATE vs git, ~15 lines |
| `/roadmap`   | Strategic one-page view â€” PRD phases + sprints remaining, ~25 lines |
| `/idea`      | Parking-lot capture to `docs/local/ideas.md` |
| `/replan`    | Structured PRD revision with Revision Log entry + completion-plan regen |
| `/ship`      | Ship protocol â€” quality gate, push, merge, checkpoint, distil progress to project-log, cleanup |
| `/log`       | Milestone writer â€” appends one canonical line to `docs/local/project-log.md` |

Each command is a top-level skill at `.claude/skills/<name>/SKILL.md`, living as a sibling of
`project-log/` so Cowork exposes it as a real slash command. Templates they read live at
`.claude/skills/project-log/templates/`.

**Description-driven dispatch.** Every sibling SKILL.md description names the slash command **and**
lists the natural-language phrasings that should fire it. Claude is expected to pick the right
skill from context; the user does not have to remember which slash command does what.

## Tool-stack abstraction

`docs/sprint-loop.md` has a YAML frontmatter block the skill reads to adapt to any stack. See
`templates/sprint-loop.md` for the canonical shape. Edit the frontmatter; leave the body alone.

## What this skill is NOT

- Not a ticket tracker (no Jira, Linear, GitHub Issues sync)
- Not a team product (one user, one repo)
- Not a code-quality tool (that's lint/tests, configured via sprint-loop frontmatter)
- Not a strategic advisor â€” it won't tell you what to build, only help you stay on plan
